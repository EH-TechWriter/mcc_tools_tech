Progressive Information Disclosure

What is it?

The right amount of information available at the right time.

As a user interacts with a thing, they can find the information they need when they need it.

A many layered onion

  1. The user interface
  2. Hover text
  3. More information ( or )
  4. Learn more...
  5. Tours
  6. Help

The user interface

Many people believe that "Technical Writing" is more about User Experience and Information Design

Some folks believe tech writers should own all the text on the screen.

Clear labels and guidance text help a lot.

There's nothing worse than documenting around poor UI design.

UI Text









See how a little bit of UI text can save you some documentation?

Hover text

Also called "Tool Tips"

(Sometimes used with Accessibility, too)

When you hover (maybe click, depending) over content, additional text appears in a popup.

Currently not available on mobile devices.

More information

Often indicated with or

Click (or hover) over the icon and get a blurb of text.

The information tends to be longer than a pop up/hover help.

This can be in a popup near the icon, or maybe open a slide out panel or another screen.

Learn more

At the end of a pop-up, you might want to include a Learn more... link.

A link to more in-depth information.

This can open up the help, a slide out panel, or go off to other web sites.

Tours

Those bubbles in Facebook or Adobe Acrobat highlighting new features.

These text bubbles step you through some task in the UI.

Some are interactive, some are just informational.

Help

The user manual integrated into your software.

When possible, context-sensitive is better.

Context-sensitive = Click help, get information about the screen you are on.

Once upon a time, F1 was used to launch the help.

Help implementations

Over the years, the definition of "help" has changed.

  • Static PDF
  • Frameset of content with TOC, Index, Search
  • Text in a slide out panel
  • Link to content hosted somewhere

Not just in the UI

Remember, as a writer, you are a user advocate

You may need to support many types of users.

Can you show/hide information based on the user's role?

Don't fall into the trap of documenting ALL THE THINGS

More Git

Because, yes, it's tricksy

GitHub process - Forking

GitHub is designed for disparate people to share projects.

  • Person A hosts code in GitHub in a repository (repo)

    There can be multiple branches in this repo.

  • Person B wants to work on this project. Person B forks the project.

    There is now a copy of A's code in B's project in GitHub (up in the cloud)

GitHub UI

There are many, many things you can do in GitHub in the browser. Only focusing on Code and Pull Request

The Code tab has all the files in the repo. You can change the branch.

You can browse all the files and folders, view their history, and for text files, edit them in the cloud/browser.

You can browse all the commits to see what's happened recently.

The Pull Request tab lets you work with existing pull requests to review and merge them.

GitHub Cloning

While you can edit text files in the browser, doesn't mean you should.

If you download the files, all you get are a copy of the files on your local system. There's no connection between local files and the files in GitHub.

If you clone the files, you can:

  • Pull changes from the original repo (A) to your local copy.
  • Pull changes from the forked repo (B) to your local copy.
  • Push changes from your local copy to your forked repo.

Working with GitHub

This assumes you've cloned your forked repo to your local system.

  1. Make changes to your files. Add and Commit to Git. (local)
  2. Push to your forked repo. (Get into GitHub)
  3. Pull (Fetch) (Sync) changes from the original repo to your repo.
  4. Make a pull request to get your changes into the original repo.
