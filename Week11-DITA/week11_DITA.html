DITA

Writing with Legos

Building blocks

  • Easily mix and match topics in ditamaps

    A "bookmap" (PDF) and a "map" for HTML/Help output

  • Easily have shared/common topics

    Legal notices, contacting support, etc.

More building blocks

  • Small chunks that can be translated in small chunks

  • Smaller blocks possible to reuse specific paragraphs, words, chunks of content

Extensible

  • If you want to get fancy, you can create specializations

    The task, concept, and reference topics are all specializations of the generic topic type

  • You can transform DITA into anything

    All you need is the code to support it

    This includes: Presentation formats, animations/videos, flavors of wiki formats, messages to be used in UIs, etc.

Modifying blocks

  • <titlealts>

    You can change the title of your topic via the ditamap, have one title in the topic, and a different in the table of contents, and a still different one in search results.

    For example, shorter title in the TOC, but make sure you include the product name in search results

More modifying blocks

  • With keys, you can easily change references in the same content based on ditamap

    Super useful for product names, especially for different components with a shared installation guide

    For image references, can have high res for PDF and lower res for web

Still more modifying blocks

  • With conref push, you can insert information into topics

  • With ditaval, you can filter your output and have conditional text

Open standard

  • Technically, you can use DITA for free

  • Technically, if you author in DITA, anyone else can use your source to produce output (Exactly the same way all browsers render the HTML standard the same way)

  • As more and more people adopt DITA, there are more and more tools that make it more accessible. You don't have to be an XML, XSLT, JS, JAVA, ANT, and DITA expert to use it with oXygen Author, FrameMaker, etc.

Why DITA?

Small topics are building blocks to reuse

Many, many features to enable more reuse

Open source standard that can "easily" be used elsewhere

Extensible

Really, really powerful

Why Not

  • DITA is very rich and featured, therefore complicated

    DITA can do all the things...if you have the people or tools to make it work

  • Beware of DITA Tools that 'augment' your DITA

    Some CMS help create index term libraries, conrefs, etc. in proprietary formats that are no longer sharable.

Word to the wise

Just becaused you can doesn't mean you should

There's probably at least 3 ways you can do the thing you want to do in DITA. All of them will work, some will work better than others, some will provide more gray hairs than others.

DITA basics

XML file format

The DITA standard fulfills two roles:

  • Standardized file formats
  • Authoring standard

DITA files

All content must be valid.

DITA has a set of rules about what should be in a file.

Intention is to provide standarization and guidance when authoring.

Like a form, but not quite as restrictive.

DITA Standard

Based on Structured Authoring.

Three major topic types:

  • Concept - About a thing
  • Task - To do the thing (a procedure)
  • Reference - List of stuff about thing (often tables, definitions, etc.)

Topics

In general, one file equals one topic.

One topic generally equals one idea.

Concept

About a thing.

Provides very little structure, because concepts are amorphous things.

Use to write about all the background information you need to understand a thing.

Task

Procedure to do the thing.

The most structured DITA topic type.

Prerequisite, Context, Steps/steps-unordered, Result, Example, Post-requisite

Reference

Stuff you want to look up about a thing.

  • section - for just text
  • example - just text, but different intent
  • table
  • properties - like a table but more specific
  • refsyn - programming/syntax reference

Troubleshooting

A specialization for trying to fix things yourself

  • condition - what the problem is
  • troubleSolution - a container
    • cause - why the problem happens
    • remedy - how to fix or workaround the problem

Semantic tagging

Instead of applying styles, take information about what it is.

Instead of worrying about formatting, just tag information for what it is. Let the publication tools worry about formatting.

Avoid using <b> and <i>.

Work with your editors to determine how to tag items using <uicontrol>, <wintitle>, <parmname>, <codeph>, etc.

Conrefs and Keys

The next phase of reuse.

Use the @conref attribute to reuse almost any DITA element.

Conrefs only work with like elements. To reuse a <note>, you have to insert a <note>

Conrefs are associated with files.

Use the @keyref for a key, most commonly used for short words or phrases or paths to files.

Keys are associated with maps.
